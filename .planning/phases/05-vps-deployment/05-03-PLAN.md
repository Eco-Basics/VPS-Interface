Critical design constraints from CONTEXT.md (NON-NEGOTIABLE):

- Uses `pm2 reload claude-vps-interface` (not `pm2 restart`) — zero-downtime

Exact contents:

```bash
#!/usr/bin/env bash
set -euo pipefail

# deploy.sh — Update the Claude VPS Interface to the latest git commit.
# Run this script on the VPS: SSH in, then run ./deploy.sh
#
# What this does:
#   1. Pull latest code from git
#   2. Install/update dependencies (npm ci — reproducible, uses package-lock.json)
#   3. Compile TypeScript (npm run build → dist/)
#   4. Zero-downtime reload via pm2 (new process starts before old one dies)
#
# What this does NOT do:
#   - Touch .env (secrets are set-and-forget)
#   - Run database migrations (no database in this project)
#   - Restart Caddy (Caddy config is separate)

APP_NAME="claude-vps-interface"

echo "=== Deploy: $APP_NAME ==="
echo ""

echo "[1/4] Pulling latest code..."
git pull

echo "[2/4] Installing dependencies..."
npm ci

echo "[3/4] Building..."
npm run build

echo "[4/4] Reloading pm2 process (zero-downtime)..."
pm2 reload "$APP_NAME"

echo ""
echo "Deploy complete. Check status with: pm2 status"
```

After writing the file, make it executable:
  chmod +x deploy.sh

The `set -euo pipefail` at line 2 ensures that if any step fails (e.g., build error), the script exits immediately and pm2 reload is never reached — the running process stays up.

```markdown
# Caddy Setup Guide

Caddy provides HTTPS reverse proxy. It listens on port 443 (HTTPS) and port 80 (HTTP → redirects to HTTPS), proxying all traffic to the Node.js app on localhost:3000.

No domain name is required. Caddy generates a self-signed certificate using its internal CA (`tls internal`). Your browser will show a security warning on the first visit — add a browser exception once and it will not appear again for that browser.

Traffic is still TLS-encrypted end-to-end.

## Caddyfile

Replace the default `/etc/caddy/Caddyfile` with:

```
:443 {
    tls internal
    reverse_proxy localhost:3000
}

:80 {
    redir https://{host}{uri} permanent
}
```

Access the app at: `https://<your-vps-ip>/`

## Applying the Config

Edit the Caddyfile:

    sudo nano /etc/caddy/Caddyfile

Paste the contents above. Then validate and reload:

    # Validate config syntax
    sudo caddy validate --config /etc/caddy/Caddyfile

    # Reload without downtime
    sudo systemctl reload caddy

    # Verify Caddy is running
    sudo systemctl status caddy

## First-Time Browser Exception

When you first open `https://<your-vps-ip>/`:

1. Chrome/Edge: Click "Advanced" → "Proceed to <ip> (unsafe)"
2. Firefox: Click "Advanced..." → "Accept the Risk and Continue"
3. Safari: Click "Show Details" → "visit this website"

You only need to do this once per browser. The certificate is generated by Caddy's internal CA and is self-signed.

## Troubleshooting

**502 Bad Gateway** — Node.js app is not running on port 3000. Check: `pm2 status`

**Connection Refused** — Caddy may not be running. Check: `sudo systemctl status caddy`

**Port 443 already in use** — Another service is on port 443. Check: `sudo ss -tlnp | grep :443`

## Caddy Management

    # Check status
    sudo systemctl status caddy

    # Reload after config change (no downtime)
    sudo systemctl reload caddy

    # View Caddy logs
    sudo journalctl -u caddy -f

    # Restart (if reload is insufficient)
    sudo systemctl restart caddy
```
